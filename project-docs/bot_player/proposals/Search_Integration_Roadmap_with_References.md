# Global Conquest – Search Integration Roadmap

> Elaborates milestone 11 ("Search integration") of `GCN_Strategy_Roadmap_with_References.md`, now that milestones 1-10 (schema, encoding, candidate afterstates, supervised GCN, TD(λ) training) are built and validated. A deeper-but-still-single-opponent adversarial-ply lookahead (`LookaheadDepth` on `ValueStrategy`) was built, evaluated, found to make things worse, and removed from the codebase — see "Motivating result" below for the actual numbers and analysis, preserved here even though the code itself is gone. Describes what it actually takes to move from that to the paper's real search-based Attack Handler.
>
> **Status: Phase 2 (move-sequence search) implemented and unit-tested, with minimal breadth pruning (`AttackSearchBreadth`) and four real, unrelated infrastructure bugs found and fixed before valid results could exist (`AttackTerminalStates`' memory-complexity blowup, the simulation harness's stall-detector false positive, `ForecastAttack`'s missing cross-call caching). Depth=2 (n=200): 26.4% on the killbot-free matchup (real improvement vs. ~17% baseline), 16.2% on the killbot matchup (flat vs. baseline, but a real rank change from last to second place). Depth=3 (n=100, killbot matchup): 10.3%, no better than depth=2 at ~3x the cost — pushing raw depth further has been abandoned; depth=2 is the working configuration going forward. The project's stated bar is outperforming `killbot-v1` (dominant at 58-72% across runs) as a milestone toward genuine competitiveness (killbot itself is easy for a human to beat — this is not the finish line) — not yet met.** This is a living record, not a pre-work snapshot — updated as each phase actually happens, so it can double as source material for a project retro once a competitive bot exists.

## Primary References

1. **Using Graph Convolutional Networks and TD(λ) to Play the Game of Risk** (Jamie Carr, 2020) — https://arxiv.org/abs/2009.06355. Section 3.1 (search algorithm), Section 3.3 (Attack Handler). The single source of truth for what "the real thing" looks like; every design decision below either matches it or explains why we're deliberately diverging.

## Where we are today

- `internal/bot/strategy_value.go`'s `ValueStrategy.attack()` enumerates every legal attack via `risk.LegalAttacks`, scores each **independently** via a plain 1-ply `attackAfterstateBlend`, and picks the single best-scoring candidate. Non-attack phases (reinforce/occupy/fortify) never look ahead at all. A deeper, still-single-opponent adversarial-ply generalization (`LookaheadDepth`, `internal/bot/lookahead.go`) was built and evaluated this session, then removed after measuring worse — see "Motivating result" below. `attack()` itself is still untouched by real search as of Phase 1 — `internal/bot/attack_handler.go` exists standalone but nothing calls it yet.
- The GCN value function (`internal/bot/gcnmodel`, trained via `analytics/.../gcn_fit.py`'s `fit_gcn_td`) reached a real, reproducible ~17% win rate this session (12 epochs, TD(λ), `--percentile 0` margin calibration) — no longer the weak link it was when the original shallow lookahead was tested and showed no improvement.
- Non-determinism (dice outcomes) is handled today via `internal/bot/afterstate.go`'s `attackAfterstateBlend`: a probability-weighted blend of two hypothetical afterstates ("conquered," "held"), not an exact simulation of any one dice sequence. This is an intentional simplification carried from the very first `BoardValue` work (see `11_Learned_Board_Evaluation.md`) and is one of the pieces the real Attack Handler replaces.

## Motivating result: the narrow greedy-chained version was tested, and made things worse

This session built `LookaheadDepth` on `ValueStrategy`: alternating "our candidate attack, then the single most relevant opponent's own greedy-best reply" for `N` plies, recursively. It was deliberately narrow in two ways this roadmap exists to remove: **(a)** scoped to exactly one opponent — the contested territory's former owner — never the whole board, and **(b)** at every ply, each side's "move" was *given* by greedy 1-ply selection, never itself explored as a branching choice.

Evaluated at depth=2 against the 12-epoch/no-lookahead baseline (~17% win rate, reproduced three times), with margins properly recalibrated for the depth=2 score distribution via `cmd/bvcalibrate --lookahead-depth` (a first attempt reused the 0-ply-calibrated margins by mistake and showed an even worse, confounded result — recalibrating first was not optional). Real, apples-to-apples result, 200 games per matchup:

| matchup | no lookahead | depth=2 |
|---|---|---|
| `board-value-candidate,boscoe-v1,killbot-v1,quo-v1` | ~17% | **5.9%** [2.5-13.0] |
| `angry-v1,basic-v1,cluster-v1,pixie-v1,scored-v1,board-value-candidate` | ~17% | **10.5%** [6.7-16.2] |

Depth=2 was worse in *both* matchups, not just flat. Three likely reasons, all of which point at the same conclusion — this isn't real search, it's a chain of greedy 1-ply picks wearing a search-shaped hat, and that shape actively hurts once it goes more than one ply deep:

1. **Errors compound instead of cancel.** Each ply commits to a single greedy-best move (`bestOwnReply`) and the next ply reasons on top of that fixed, possibly-wrong choice. Real search backs up the best of many explored branches; this only ever explores one path per ply.
2. **Afterstates are already approximations, chained on top of each other.** Every ply scores the probability-blended "conquered/held" afterstate (`attackBranches`/`blendFeatures`), not a real board state. Three or four plies deep, the leaf being scored is an approximation of an approximation of an approximation — exactly the kind of compounding synthetic-state error the real Attack Handler (below) is built to avoid by working with actual terminal-state probabilities instead.
3. **Single-opponent scope gets less realistic the deeper it goes.** Ply 1 asking "how does the territory's former owner react" is a reasonable simplification. Ply 3 asking "and then the *same single* opponent again, with the rest of the board assumed inert" is an increasingly narrow fiction.

This is the concrete evidence behind this roadmap's premise: the gap isn't "make the existing lookahead deeper," it's the five structural pieces below, each of which addresses one of the three reasons above.

## What the paper actually does (Section 3.1, 3.3, 3.5, 3.6, 3.7.1)

Re-read directly from the source PDF (not summarized memory) to get the exact mechanics right before building anything.

- **The core loop is search, not greedy scoring.** A heuristically-pruned breadth-first search over the game tree, re-run from scratch after *every single attack* (not once per turn) — because a single attack's dice outcome changes the tree. The GCN value network only ever scores *leaves* of this search; it is never used to greedily rank a flat list of candidates the way `ValueStrategy.attack()` does today.
- **Search is time-budgeted, not depth-budgeted**: a `Search Time` = 10-second-per-turn wall-clock budget, anytime-style (return the best move found so far if the clock runs out), not a fixed ply count. Real turns could take 1-2 minutes when the search actually used its full budget across multiple re-searches (one per attack) in one turn — a genuine cost to expect, not a bug to eliminate.
- **The Attack Handler (Section 3.3) is the load-bearing, non-obvious piece**, and its exact mechanism (corrected from an earlier, wrong summary in this doc):
  1. Build `G`, a tree of every reachable `(attackers, defenders, probability)` state from the starting `(A, D)`, expanding every leaf by every possible single-round dice outcome (attacker rolls up to 3 dice, defender up to 2, highest-vs-highest then next-highest-vs-remaining, ties favor the defender), merging children that land on the same `(a, d)`.
  2. Every branch that reaches a **terminal state** (paper's definition: `a=0` or `d=0` — this project's own implementation instead matches the engine's real rule, `d<=0` or `a<=1`, since the engine never allows an attacker to be reduced below 1) is removed from `G` and added to `R`, the full list of terminal states, ordered **best-for-defender → best-for-attacker**.
  3. To pick one deterministic outcome to continue searching from: walk `R` **from the best-for-defender (worst-for-attacker) end**, accumulating probability, and commit to the **first** terminal state where cumulative probability reaches the **`Risky`** threshold (`Risky ≤ Σp_t`). A **higher** `Risky` walks further toward attacker-favorable outcomes before committing (more optimistic); paper value **0.3**, described as balanced ("too high results in over-optimistic attacks... too low results in very passive behavior") — not an extreme value despite being numerically low, since combat probability mass tends to concentrate, so even a modest 30% slice from the pessimistic end already reaches a fairly typical outcome.
  4. Very large attacks with no precomputed table entry are calculated dynamically (the paper precomputes a lookup table up to a cap for speed; this project's `ForecastAttack`/Attack Handler already compute fresh per call and don't need this optimization unless profiling says otherwise).
- **Search heuristics bound branching per stage** (Section 3.5.1), values from 3.7.1: placing only considers territories bordering an enemy, only from `Tp=2` territories at a time, in groups of `Gp=3` armies. Attacking never considers a candidate where the enemy outnumbers the attacker, and after a successful attack only considers `Ga=3` variations (interpolated from the legal minimum of 3) for how many units to move in. **Fortification is explicitly disconnected from the main search** — evaluated by comparing every fortification against doing nothing and repeating until nothing beats it, in groups of `Gf=10` (cheap enough not to affect search time) — this is exactly what `ValueStrategy.fortify()`/`TurtleStrategy`'s fortify logic already do, a confirmed point of reuse, not a gap.
- **Why BFS, not MCTS or DFS** (Section 3.5.2-3.5.4): MCTS was tried and discarded — batching the evaluation function into MCTS's leaf-by-leaf structure gave 10-100x *fewer* states searched than BFS in the same time budget. DFS was tried and discarded — it spends most of its budget on deep, sophisticated lines and regularly misses obvious moves. BFS's own weakness (short-sighted, only a few attacks deep per search) is mitigated by re-running the whole search after every single attack, so it can still execute complex multi-attack sequences across several re-searches.
- **Two "Final Changes" after initial testing (Section 3.6)**: (a) once a player holds ≥95% of total board armies, the evaluation function is overridden to just "territories owned + 1" — a hardcoded fix for the network failing to distinguish "almost won" from "actually won" under a depth-limited search, so it would sometimes leave a nearly-eliminated opponent alive indefinitely; (b) the `Defence` feature is capped at 0.2 before normalization, since values outside the training distribution confused the network — **already implemented** in this project's `internal/tdstate/encode.go`'s `defenceCap`, confirmed matching, not a gap.
- **Exact tuned parameters (Section 3.7.1)**, as a reference table:

  | parameter | paper value | this project |
  |---|---|---|
  | Risky | 0.3 | not yet set — treat as a starting point to re-tune, not copy verbatim (see "Open design questions") |
  | λ (TD(λ)) | 0.8 | already matches (`fit_gcn_td`'s `lam`) |
  | Defence cap | 0.2 | already matches (`defenceCap`) |
  | GCN1/GCN2/FC1/FC2/FC3 out | 60/30/60/60/30 | already matches (`GCNValueNetwork`) |
  | Tp / Gp / Ga / Gf | 2 / 3 / 3 / 10 | not yet used (Phase 4) |
  | Epochs | 3 | our own independently-found sweet spot for `fit_gcn_td` was **12**, not 3 — a real, measured difference, not just an unset default (see the TD(λ) epoch-tuning history: 3/6/10/12/18 tested, 12 was the reproducible peak) |
  | Optimizer / learning rate | Adadelta, α=0.5 | plain SGD, `alpha=1e-3` — not directly transferable, different training setup (see `fit_gcn_td`'s own docstring for why Adam/Adadelta-style adaptive optimizers caused collapse here) |
  | Search Time | 10s | not yet set (Phase 5) |

- **Actual paper results, as a concrete benchmark**: 283 games vs. the hardest inbuilt Lux AIs (Killbot, EvilPixie, Bort — none in its own training data) → **35.3% win rate**, nearly double Killbot's ~18%. Worth keeping in mind: even the paper's own "success" is a 2x-baseline result, not dominance.

## Gap analysis: five things we don't have

1. **The Attack Handler itself — closed, Phase 1.** `internal/bot/attack_handler.go` now computes the full terminal-state probability distribution (generalizing `internal/bot/combat_odds.go`'s `ForecastAttack`, which only returns a summary statistic). Standalone and tested; not yet consulted by any decision-making code — see "Proposed phased approach" item 1.
2. **True multi-player branching** — considering more than one designated opponent. Both the original shallow lookahead and this session's deeper version deliberately stay single-opponent-scoped. Real search needs to consider (some principled subset of) all rivals' territories, not one.
3. **Heuristic pruning of branching factor** — not needed yet because `bestOwnReply` always greedily collapses each side's options to one before recursing (that's *why* the existing lookahead is only `O(2^depth)`, not combinatorial in candidate count). Real search that considers multiple candidates per side at each node needs actual pruning heuristics (the paper's `Tp`/`Gp`/`Ga`/`Gf`), which is a design-and-empirically-tune problem in itself.
4. **Anytime, time-budgeted search** — nothing in this codebase currently bounds a decision by wall-clock and returns a best-so-far answer. `Strategy.NextCommand`'s existing `ctx context.Context` parameter is actually a good hook for this (`context.WithTimeout` is the idiomatic Go primitive), but the search loop itself (iterative-deepening or similar, checking `ctx.Done()` and returning the best candidate found) doesn't exist.
5. **Searching over move *sequences*, not scoring pre-enumerated single moves — closed for our own attacks, Phase 2.** `ValueStrategy.attack()` now branches (`AttackSearchDepth > 0`) into a real sequence search (`attackSequenceSearch`/`bestContinuation`) instead of only ever scoring pre-enumerated single moves in isolation. Still open: reinforce/occupy/fortify remain single-candidate scoring (never in scope for this roadmap — only attack-phase search was ever planned), and multi-player branching (item 2 above) is separate, later work.

## Proposed phased approach

Each phase should be independently testable and independently valuable — matching how every lever this session (margins, epochs, Turtle's security threshold, lookahead depth) needed its own empirical validation pass, not a single big-bang integration at the end.

1. **Attack Handler — done.** `internal/bot/attack_handler.go`: `TerminalState`, `AttackTerminalStates(attackerArmies, defenderArmies int) []TerminalState`, `SelectTerminalState(states []TerminalState, risky float64) TerminalState`. Built by generalizing `ForecastAttack`'s exact recursive walk over `roundDistribution`/`diceOutcome`/`forEachRoll` from a scalar collapse to a full terminal-state distribution, with the same memoization pattern. `internal/bot/attack_handler_test.go` covers: probabilities summing to 1 across representative `(a, d)` pairs; cross-validation that summed win-outcome probability exactly matches `ForecastAttack`'s `WinProbability` (confirms the new accumulation logic against the already-tested recursion it shares); a hand-derived `a=2, d=1` case (21/36 attacker-stops, 15/36 attacker-wins); `SelectTerminalState` monotonicity across a sweep of `risky` values and its two edges. `go test ./internal/bot/... -race` and the full `go test ./... -race` both pass.

2. **Move-sequence search over our own attacks — implemented, not yet tournament-validated.** `ValueStrategy` gained two exported fields, `AttackSearchDepth int` (0 = original single-ply `attackAfterstateBlend` behavior, unchanged) and `Risky float64` (falls back to the paper's 0.3 when `<= 0`). New `internal/bot/attack_search.go`: `attackSequenceSearch`/`bestContinuation` explore **every** legal attack at **every** level up to `AttackSearchDepth` (real branching, unlike the removed `LookaheadDepth`'s single greedily-picked path), materializing each attack via `AttackTerminalStates`/`SelectTerminalState` into one concrete deterministic board state — not a probability blend — via a new `applyTerminalOutcome` helper in `afterstate.go` (sits next to `attackAfterstateBlend`, same `copyGameState`-then-overwrite-`Territories` idiom, but for one selected real outcome instead of an averaged one). `attack()` branches on `AttackSearchDepth > 0` to pick between the two; the margin gate (`clearsMargin`/`Observer`) is unchanged and applies identically to either path's `bestScore`.

   `internal/bot/attack_search_test.go`'s core test constructs a scenario (`greenlandSequenceGame`) that specifically exercises the property `LookaheadDepth` structurally couldn't have: a first move (Alaska → Northwest Territory) that captures nothing the value function rewards by itself, but is the *only* path to a territory (Greenland) that does, two attacks away. At `AttackSearchDepth=2` the search correctly picks that non-locally-best first move; at `AttackSearchDepth=1` it correctly can't see it and ends the attack phase instead — confirming both real branching (not a chain of single best guesses) and that the depth parameter actually bounds what's found, hand-verified against the exact Attack Handler probabilities the scenario's army counts produce (worked through by hand during design: Alaska(30)→target(1) and the follow-on NW-Territory(3)→Greenland(1) both resolve to a win under default `Risky=0.3`, confirmed by the test passing).

   `cmd/bvcalibrate` gained `--search-depth`/`--risky` flags so margins can be calibrated *for* a given search configuration before evaluation — applying, from the start this time, the lesson learned the hard way when `LookaheadDepth` was first evaluated with the wrong (0-ply-calibrated) margins. `cmd/tournament`'s `--board-value-variant`/`--gcn-variant` flags gained an optional `,search-depth=N,risky=R` suffix (shared parsing via `searchVariantOptions` in `board_value_variant.go`) so search-enabled variants can be registered and compared against the depth-0 baseline in an actual tournament. **Not wired into `cmd/backend/main.go`** — stays tournament/`bvcalibrate`-only until real win-rate evidence justifies promoting a specific depth to live play, same staging discipline `GCN_MODEL_PATH` itself went through.

   `go build ./... && go vet ./... && go test ./... -race` all pass.

   **A genuine O(a·d·(a+d)) memory-complexity bug was found and fixed in this function, after it shipped and passed every unit test.** Two tournament attempts (200 games each, one at `--parallel 12`, one at `--parallel 4`) both died with `signal: killed` partway through — reducing parallelism made it die *sooner* (66/200 vs 199/200), which ruled out simple concurrent-memory-pressure as the cause and prompted an actual memory profile instead of a third guess. A 40-game sequential single-process diagnostic (`runtime.MemStats` sampled after every game) found `Sys` climbing from 14MB to over 1.2GB in sharp jumps, each landing exactly on a game that was unusually slow or hit a 3-minute safety timeout, while `HeapAlloc` stayed near-zero throughout — a huge *transient* single-call spike, not a cross-game leak.

   Root cause: the original backward-recursive `AttackTerminalStates` memoized a **full terminal-state list** (up to `a+d` entries) at every one of the up to `a·d` internal `(a, d)` states it visited during one call — total memory roughly `O(a·d·(a+d))`. Every existing test used small, hand-picked army counts (2-30), where this is invisible; a real, longer game piling armies into the hundreds on one border territory (completely normal, not a contrived edge case) made a single call blow up to hundreds of MB in seconds. Fixed by replacing the backward recursion with a forward dynamic-programming sweep over the `(a, d)` grid — `O(a·d)` time and space, no `(a+d)` multiplier, same output contract (verified: every existing test, including the exact hand-computed `a=2,d=1` case, passes unchanged against the new implementation). Added `TestAttackTerminalStates_LargeArmiesStayFast` (`a=300, d=250`, asserts under 1s) as a permanent regression test — the exact gap every prior test left uncovered. Re-ran the same 40-game diagnostic that found the bug: `Sys` now stays under 80MB for the whole run, no crash.

   **Lesson for the retro**: this is a case where "add a unit test for the new function" wasn't enough — every Phase 1 test was correct and passed, but all of them used small numbers because that's what's hand-computable and what the Attack Handler's own worked examples use. The bug only existed at the scale a real game actually reaches, and was only found by testing at that scale (many full games, not synthetic single calls) with actual memory instrumentation, not by reasoning about the code or by more unit tests of the same shape.

   **First tournament attempt (200 games, depth=2, `--search-depth 2 --risky 0.3` calibrated margins) was not a valid result — a computational-limits artifact, not a win-rate measurement.** Every strategy in the matchup showed `completed: 5` out of 200 `appearances` — 195 games hit `simulation.DefaultLimits().MaxDuration` (30s wall-clock per game) before anyone won, not because search plays badly but because it's too slow to finish a game inside the default budget. Timed directly (`ValueStrategy.attack()` on a realistic 4-player mid-game state, 38 legal attacks for the acting player): depth=0/1 ≈ 70ms, **depth=2 ≈ 2.5s, depth=3 ≈ 88.6s** — roughly a ×35 jump per extra ply, consistent with the "no pruning, grows with (legal attacks)^depth" limitation called out in this doc's Phase 2 design section. A single turn typically involves several attack decisions (the search re-runs from scratch after every real attack, by design), so at ~2.5s/decision, depth=2 can burn through the entire 30s game budget in a single turn on a richly-connected board — this isn't a depth=2-specific edge case, it's the expected cost of shipping Phase 2 without Phase 4's pruning, just more severe in practice than "keep depth small" anticipated.

   **Implication for the roadmap's own phasing**: Phase 4 (heuristic pruning, the paper's `Tp`/`Gp`/`Ga`/`Gf`) was sequenced after Phase 2 on the theory that "does sequence search help" could be isolated and tested before adding pruning. That's not actually true in practice — unpruned search at any depth beyond 1 is too slow to complete enough games for a meaningful win-rate sample at all, so Phase 2's own success criterion (a real tournament comparison) can't be met without either pruning first or accepting a much smaller, slower, less statistically confident evaluation. This is exactly the kind of real, load-bearing finding this doc exists to preserve for the eventual retro, independent of which path gets chosen next.

   **Minimal breadth pruning added in response — `AttackSearchBreadth`.** The user chose to pull forward a small slice of Phase 4 rather than accept a much longer/smaller-sample tournament run or settle for the weaker depth=1-only test. Checked first whether the paper's own attack-phase heuristics (`Tp`/`Gp`/`Ga`/`Gf`) already solved this: they don't — `Tp`/`Gp` are specifically about the *reinforce* phase, and `Ga=3` bounds the *occupy* amount after a successful attack, not which attack candidate to explore deeper; the paper's only real attack-candidate filter ("never consider an attack where the enemy outnumbers the attacker") is too weak here since the blowup comes from many individually-reasonable mid-game attacks, not obviously-bad ones. So this is new, purpose-built pruning, not a paper value plugged in.

   `ValueStrategy.AttackSearchBreadth int` (zero = unlimited, matching Phase 2's original, still-tested full-enumeration behavior) caps each search level to the top-scoring candidates, ranked by the existing single-ply `attackAfterstateBlend` score — reusing the already-~17%-validated evaluation as a cheap pre-filter, on the reasoning that the sequence search is a *refinement* of that already-good choice, so only the most promising few candidates are worth the expensive deeper look. New `candidateAttacks` helper in `attack_search.go` applies the cap uniformly at every level (including the top) since the top level's own branching is exactly as much of the cost problem as any deeper level.

   Re-timed after the fix (same realistic mid-game state, 38 legal attacks): **depth=2/breadth=5 ≈ 400ms/decision (down from 2.5s unpruned), depth=2/breadth=8 ≈ 620ms, depth=3/breadth=5 ≈ 1.8s, depth=3/breadth=8 ≈ 4.7s** — depth=2 at breadth≈5-8 is now comfortably tournament-testable inside the default 30s/game budget; depth=3 likely still needs a smaller breadth or a raised `--max-duration` for the same reason.

   `internal/bot/attack_search_test.go` gained a test that documents the real tradeoff explicitly (not just asserts speed): a scenario where breadth=0 finds a 2-ply sequence to a large reward, but breadth=2 prunes out the one first move that leads there (because that move's own 1-ply score is smaller than two other, dead-end moves' scores) and settles for a smaller, immediately-visible reward instead — pruning can hide the genuinely-best sequence, and that's a real, worth-knowing cost of the speedup, not a hidden one. `cmd/bvcalibrate --search-breadth` and `cmd/tournament`'s `,search-breadth=N` variant suffix (both alongside the existing `search-depth`/`risky`) let this be calibrated and A/B-tested the same way as the other search parameters. `go build ./... && go vet ./... && go test ./... -race` all pass.

   **Second and third tournament attempts, after breadth pruning, both crashed (`signal: killed`) rather than producing a win-rate result** — root-caused to the `AttackTerminalStates` memory-complexity bug described above (found via those crashes, not before them) and now fixed. Re-running the same 40-game sequential memory diagnostic that found the bug post-fix: `Sys` stays under 80MB for the whole run (vs. spiking past 1.2GB before), confirming the crash itself is resolved.

   **Still not fully fast, even fixed**: of those 40 games, several still took 40s-3 minutes (one hit a 3-minute safety timeout at turn 292/command 11105, two others took 1m22s and 2m48s), so `--max-duration` needs to be generous (≥180s) for a clean tournament run even post-fix — depth=2/breadth=5 is far from free, just no longer catastrophically broken.

   **A `repeated_state_detected` failure (initially misdiagnosed) turned out to be an unrelated, real bug in the simulation harness, not in the search or in `quo-v1`.** First 40-game diagnostic hit one `repeated_state_detected` failure; a 200-game tournament run (matchup `board-value-candidate,boscoe-v1,killbot-v1,quo-v1`) showed this failure mode as *dominant* (102 of 131 failures, only 69/200 completed) — significant enough to investigate properly rather than footnote. `TraceFull` on a reproduced failure (seed 4) showed the "stuck" player was `quo-v1`, repeatedly attacking `Middle East → Ukraine` for 500+ commands, phase/player never changing — first assumed to be a bug in `quo-v1`'s own attack loop (it has no margin/progress check, unlike `ValueStrategy`). **That assumption was wrong.** Reading the actual per-round army counts (via `DomainEvent`'s `CombatRollResolvedPayload`) showed a real, steadily-progressing fight: Middle East ~1670→1240, Ukraine ~643→123 over the traced window — two armies in the hundreds/thousands (accumulated over 200+ turns without either side spending them down), grinding toward a genuine eventual capture, just slowly, since the engine resolves only 1-3 armies per dice round. `quo-v1`'s decision to keep attacking was entirely rational and effective, not a bug.

   The real root cause was `internal/simulation/limits.go`'s `MaxCommandsWithoutProgress` (default 500): it treats "same `CurrentPlayer`/`Phase` for N commands" as a stall, but that's a coarse proxy that doesn't hold once armies reach the scale a 200+-turn game can produce — hundreds of legitimate same-phase dice rounds are entirely normal at that point. `MaxRepeatedStates` (exact state-fingerprint recurrence, default 3) is the mechanism that actually detects genuine non-progress precisely, since real combat changes the fingerprint every round; it was never the one misfiring. Fixed by raising `MaxCommandsWithoutProgress`'s default from 500 to 5000 (`internal/simulation/config.go`), with the doc comment updated to record why the old value was miscalibrated. Not scoped to search specifically — this was a pre-existing harness limitation that any long enough game, with or without search, could eventually trip; the search work's longer per-decision cost just made 200+-turn games (and therefore this failure mode) more likely to actually get exercised in a tournament run.

   **Lesson for the retro**: the first read of this failure (misattributing it to `quo-v1`'s decision logic) came from pattern-matching "this bot keeps doing the same thing" without checking what was actually happening underneath — reading the real army-count trajectory took one more diagnostic pass but completely changed the diagnosis and the fix. Worth remembering next time a strategy "looks stuck": check whether the underlying state is actually static before assuming the decision logic is at fault.

   **Caveat on the fix, flagged by the user directly**: army stacks in the hundreds/thousands after 200+ turns aren't realistic in human terms — real Risk players attack far more readily than sit on accumulated armies, since an idle stack is wasted potential, not a defensive asset worth preserving indefinitely. Raising `MaxCommandsWithoutProgress` was still the correct fix (a real, in-progress game state should never be misclassified as a harness bug), but it accommodates the *symptom* rather than asking why games are reaching this scale at all — which points at a separate, likely more consequential question: are the bots in these matchups (any of them, not just the search-enabled one) too passive/turtling relative to how a human would actually play, spending many more turns reinforcing than attacking? If so, that's plausibly also relevant to why `board-value-candidate` tops out at 26-30% rather than dominating — a hypothesis for later investigation, not yet checked.

   **Both standard matchups now have real, trustworthy tournament results (n=200, depth=2/breadth=5/risky=0.3) — a split outcome, not a clean win.**

   - `angry-v1,basic-v1,cluster-v1,pixie-v1,scored-v1,board-value-candidate` (no killbot-v1): **26.4% win rate [20.5-33.2 95% CI]**, second place behind `pixie-v1` (47.3%), vs. a ~17% depth=0 baseline on this matchup. CI lower bound sits comfortably above 17% — a real, statistically meaningful improvement, not noise.
   - `board-value-candidate,boscoe-v1,killbot-v1,quo-v1` (the matchup that actually matters for the stated "beat killbot" goal): took two attempts to get a trustworthy number. First attempt (`--max-duration 200s`) completed only 128/200 games and showed 10.2% — but that specific matchup has a much lower completion rate than the other one, and the exact same selection-bias mechanism suspected in the original crashed run (games where the search bot is competitive run longer, more likely to hit the duration cap, more likely to be excluded) was still live at that scale. Re-run at `--max-duration 500s` completed 154/200 and the win rate rose to **16.2% [11.2-22.9 95% CI]** — confirming the bias was real (completion rate and win rate both rose together) and landing almost exactly on the ~17% baseline. `killbot-v1` remained dominant throughout (58.4-64.8% across both attempts).

   **Net read against the stated goal**: search helps in the matchup without killbot-v1, and is statistically indistinguishable from the depth=0 *baseline* in the matchup that has it — **the actual goal (outperform killbot-v1) is not yet met.** But a sharper, more useful way to state the killbot-matchup result (per the user's own read, confirmed by comparing the two attempts directly) isn't "neutral" — it's a real rank change: `board-value-candidate` went from *last place* (10.2%, behind both `boscoe-v1` at 11.7% and `quo-v1` at 13.3%) in the biased low-completion run to clearly *second place* (16.2%, ahead of `boscoe-v1` at 12.3% and `quo-v1` at 13.0%) once the completion-rate bias was fixed — while `boscoe-v1`/`quo-v1` themselves barely moved between runs. So search-enabled play is genuinely stronger than the other heuristic bots in this field; it just hasn't caught `killbot-v1` (58.4%, still dominant) yet. Whatever comes next (Phase 3 multi-player branching, Phase 4 fuller pruning, hyperparameter tuning) should be evaluated specifically against the killbot matchup, not the easier one, per the user's explicit framing of the goal — and the bar to clear is specifically killbot's ~58-65%, not just "above the depth=0 baseline."

   **Important, also stated directly by the user: beating killbot-v1 is a milestone, not the finish line.** `killbot-v1` is easy for a human to beat — it's just the strongest opponent currently implemented in this codebase. The real goal is a bot that's genuinely competitive. This project has no human-benchmark evaluation mechanism yet; that becomes the natural next open question once/if the killbot milestone is actually reached, not something to solve now.

   **`ForecastAttack` caching fix — depth=3 is now viable.** User chose "push search harder" as the next step; testing depth=3 (chosen since raw breadth increases at depth=2 are comparatively cheap and don't need new engineering, while depth=3 is the more meaningful sophistication increase) immediately hit a wall: a single real decision failed to complete within a 30-minute timeout. Root cause, found by reasoning about the algorithm's own structure (not more blind profiling): at depth=3 the search visits 31 tree nodes (1 + 5 + 5×5 for breadth=5), and at *every* node, `candidateAttacks`' ranking pass scores *every legal attack* at that state (uncapped by breadth — you need to score everything to know the top ones) via `ForecastAttack`, whose cost scales with army-count size. With armies reaching into the hundreds/thousands in long games (established earlier), and no caching *across* the ~1000+ ranking calls one decision makes, the redundant recomputation compounded into a real multi-minute-plus cost — the same underlying issue class as the `AttackTerminalStates` bug, surfacing in a different function this time.

   Fixed with a new `forecastCache` (`internal/bot/attack_search.go`): a `map[[2]int]CombatForecast` created once per `attack()` decision and threaded through `attackSequenceSearch`/`bestContinuation`/`candidateAttacks`, so every `ForecastAttack`-equivalent call within one decision's whole search tree shares results instead of each of the ~1000+ ranking calls re-deriving them. `attackAfterstateBlend` gained a generalized `attackAfterstateBlendWithForecast(g, pi, a, forecastFn)` sibling so the cache could be injected without changing `attackAfterstateBlend`'s own signature/behavior for its many other (non-search) callers — confirmed via the full existing test suite passing unchanged. Deliberately scoped to one decision (not global/cross-game) — no concurrency concerns, bounded by one search tree's distinct `(a, d)` pairs, not process lifetime.

   Result: the exact same depth=3 full-game test that failed to finish in 30 minutes now completes in 5-10 minutes (two of five sample games instead hit the unrelated, pre-existing `MaxCommands=20000` safety net, itself reached in under a minute — a different, harmless limit, not a performance problem). In this small 5-seed sample, `board-value-candidate` won 2 of 3 completed games, including one in the exact killbot matchup — encouraging, but nowhere near a real sample size yet.

   **Depth=3 tournament result (n=100, killbot matchup, `--max-duration 900s`): not an improvement over depth=2, and pushing raw depth further was abandoned as a strategy as a result.** 10.3% win rate [5.1-19.8 CI], `killbot-v1` even more dominant (72.1%, up from 58.4% at depth=2). Completion rate was 68% (32/100 missing — split between `command_limit_reached`, a *new* failure mode at this depth, and `duration_limit_reached`), so the same selection-bias risk that affected depth=2's first attempt can't be ruled out here either — the CI is wide enough to overlap both the baseline and depth=2's result, so this isn't a statistically clean "depth=3 is worse," just "depth=3 shows no sign of being better, at roughly 3x the wall-clock cost per tournament (2h18m for 100 games vs. depth=2's runs) and a new, more severe failure mode from games running even longer (avg 199.9 turns, ~8000 commands even among completed games)." Properly bias-correcting this result would need another multi-hour run for a number that, even in the best case, doesn't look like it closes the killbot gap. **Decision: stop pushing raw depth further — depth 2→3 shows diminishing-to-negative returns at rapidly increasing cost. Falling back to depth=2 as the working configuration; the next lever should be something other than "more depth."**
2. **Move-sequence search over our own attacks — implemented, not yet tournament-validated.** See "What the paper actually does" above and `internal/bot/attack_search.go`/`ValueStrategy.AttackSearchDepth`. No multi-player branching yet — still single-opponent-scoped at the leaves, isolating "does sequence search over our own moves help" as one variable, per this item's original scoping.
3. **Multi-player branching**: broaden from one designated opponent to a principled subset of rivals (e.g. every player with a territory adjacent to the contested region) at reply nodes.
4. **Heuristic pruning**: once multiple candidates per side are genuinely explored (not greedily collapsed to one), bound branching with pruning heuristics analogous to the paper's `Tp`/`Gp`/`Ga`/`Gf`.
5. **Time-budgeted anytime search**: wrap the whole tree walk in a `context.WithTimeout`-aware loop returning the best move found if the deadline hits — replacing the fixed-depth cutoff entirely. **Scoping note from the user**: slow per-decision cost is fine for actual live human-vs-bot play — humans take their time on their own turns too, so a bot thinking for seconds (even up to the paper's own 10s Search Time, or longer) is not a UX problem the way it is for *tournament/automated-evaluation throughput*, which is the actual reason speed has mattered so much in this phase (200-game batches needing to finish in reasonable wall-clock time, not degrading live play). This phase's motivation is evaluation-loop speed, not live-play responsiveness — don't let that distinction blur when prioritizing future performance work.
6. **Empirical tuning pass**: Risky threshold, pruning cutoffs, time budget vs. depth tradeoffs — re-run through the same `cmd/bvcalibrate` + the two standard tournament matchups used throughout this whole session. Expect this to be its own multi-round effort, not a single run, based on how every other lever this session needed 3-6+ tuning iterations before finding a real sweet spot.

## Open design questions (decide when each phase actually starts, not now)

- Is the paper's Risky=0.3 threshold, or its Tp/Gp/Ga/Gf pruning values, even a reasonable starting point for our board/value-function, or should these be empirically swept from scratch the way every other hyperparameter this session was (margin percentile, training epochs, `turtleSecurityThreshold`)? Lean toward the latter given this session's own track record.
- How many rivals does "a principled subset" (Phase 3) actually mean — every adjacent player, or some bounded top-K by threat?
- Does move-sequence search (Phase 2) apply only within a single turn's attack phase, or does "sequence" ever need to span the reinforce → attack → fortify boundary within one turn?
- Should search-generated leaf evaluations feed back into training data (closing the loop, generating stronger self-play data than the current heuristic personas) — this connects to the parallel `Monte_Carlo_Evaluator_Roadmap_with_References.md`'s own milestone 10 ("search-generated training targets") and may be worth coordinating rather than treating as fully separate initiatives.

## Success criteria

**The near-term bar, stated directly by the user: build a bot that outperforms `killbot-v1`, not just one that shows a positive win-rate delta somewhere.** This isn't a new, separate goal — it's the same benchmark the source paper itself used (Section 3.8: 35.3% win rate vs. the hardest inbuilt Lux AIs, "nearly double Killbot's ~18%" — beating Killbot specifically was the paper's own headline result, not an average-case improvement). Current status against this bar, with a trustworthy (bias-checked) n=200 result: `killbot-v1` remains dominant (58.4%) and `board-value-candidate` lands at 16.2% [11.2-22.9 CI], statistically indistinguishable from the ~17% depth=0 baseline — **not met.** Depth=2/breadth=5 search is a real improvement in the easier, killbot-free matchup (26.4%) but a wash in the one that actually matters for this goal.

**Important, also stated directly by the user: beating `killbot-v1` is not the actual finish line.** `killbot-v1` is easy for a human to defeat — it's the strongest opponent *available in this codebase*, not a stand-in for genuine competence. The real goal is a bot that's truly competitive, which is a higher bar than "beats the strongest bot currently implemented here." Treat clearing the killbot milestone as one waypoint, not proof of success — this project has no human-benchmark evaluation mechanism yet, which becomes the natural next open question once/if killbot is actually beaten.

Same methodology as everything else this session: a real, reproducible movement in win rate against the same two standard tournament matchups (`board-value-candidate,boscoe-v1,killbot-v1,quo-v1` and `angry-v1,basic-v1,cluster-v1,pixie-v1,scored-v1,board-value-candidate`), not an offline/theoretical metric. Given how much effort went into finding that the *training objective* and *margin calibration* — not search — were the actual blockers behind the original 0% win rate, treat "search helps" as a hypothesis to validate cheaply per-phase, not an assumed conclusion. But the `boscoe-v1,killbot-v1,quo-v1` matchup specifically is the one to keep pushing on, since it's the only one of the two standard matchups that actually contains the opponent the project is trying to beat.
